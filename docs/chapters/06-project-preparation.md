完成本章節後，PM 將能夠：

- 🚀 建立與開發團隊協作的高效工作模式

## 🎯 PM 專用快速上手

### 為什麼 PM 需要程式碼分析能力？
作為產品經理，您經常需要：
- **理解技術債務**：評估現有系統的技術狀況
- **評估開發估時**：基於程式碼複雜度做出合理判斷
- **溝通技術方案**：與開發團隊進行有效的技術討論
- **撰寫技術文檔**：為產品功能建立清晰的技術說明

**本章將讓您在30分鐘內具備基礎的程式碼分析能力！**

## 🛠️ 環境設定：Cursor Rules 配置

### 第一步：建立 PM 專用的 .cursorrules 檔案

在您的專案根目錄建立 `.cursorrules` 檔案：

```typescript
// PM 專用 Cursor Rules 配置
// 專為產品經理設計的程式碼分析設定

# 角色設定
您是一位專業的產品技術分析師，協助產品經理理解程式碼和技術架構。

# 回應風格
- 使用產品經理易懂的語言，避免過度技術術語
- 重點關注業務影響和產品價值
- 提供可操作的建議和結論
- 用項目符號和表格整理資訊

# 分析重點
## 業務邏輯分析
- 識別核心業務流程
- 分析用戶使用路徑
- 評估功能完整性
- 找出潛在的產品優化點

## 技術評估
- 評估開發複雜度（簡單/中等/複雜）
- 識別技術風險點
- 評估維護成本
- 分析擴展性

## 文檔生成
- 生成產品功能說明
- 建立 API 使用指南
- 撰寫技術需求文檔
- 製作團隊協作指南

# 輸出格式
請按以下結構組織回應：

## 📋 快速總結
- 核心功能：[1-2句話說明]
- 複雜度評估：⭐⭐⭐ (1-5星)
- 建議行動：[具體建議]

## 🎯 業務價值分析
[從產品角度的分析]

## ⚙️ 技術實作概述
[簡化的技術說明]

## 💡 PM 行動建議
[具體的後續步驟]

# 特殊指令
當我說 "PM分析" 時，請專注於產品和業務層面的分析
當我說 "技術評估" 時，請提供技術複雜度和風險評估
當我說 "生成文檔" 時，請產生適合的產品文檔
```

## 🤖 PM 程式碼分析標準流程

### 階段一：快速概覽（5分鐘）

#### 1.1 專案結構掃描
使用以下 prompt：
```
請幫我分析這個專案的整體架構：

@folder

PM分析：
1. 這個專案的主要功能是什麼？
2. 有哪些核心模組？
3. 從產品角度看，哪些是最重要的功能？
4. 評估這個專案的複雜度等級
```

#### 1.2 關鍵檔案識別
```
請幫我找出這個專案中最重要的5個檔案，並說明：
1. 為什麼這些檔案重要？
2. 它們負責什麼業務功能？
3. PM 應該重點關注哪些部分？
```

### 階段二：業務邏輯分析（15分鐘）

#### 2.1 核心功能分析
對每個重要檔案使用：
```
PM分析這個檔案：

@file:重要檔案名稱

請告訴我：
1. 這個檔案實現了什麼業務功能？
2. 用戶如何與這個功能互動？
3. 這個功能在整個產品中的重要性？
4. 有什麼潛在的產品優化機會？
```

#### 2.2 資料流程理解
```
請分析資料在這個系統中是如何流動的：

@folder

重點說明：
1. 用戶操作如何觸發資料變化？
2. 資料經過哪些處理步驟？
3. 最終用戶看到什麼結果？
4. 哪些環節可能影響用戶體驗？
```

### 階段三：技術文檔生成（10分鐘）

#### 3.1 使用文檔生成 Prompt

## 功能說明
### 主要功能點
- 功能點1：說明
- 功能點2：說明

### 用戶使用流程
1. 步驟1
2. 步驟2
3. 步驟3

## 技術要求
- 系統需求
- 相依性
- 效能考量

## 風險與限制
- 已知限制
- 潛在風險
- 緩解措施

## 後續優化建議
從產品角度的改進建議
```

## 🔧 常見問題與解決方案

### Q1: cursor rules 沒有生效？
**解決方案：**
1. 確認 `.cursorrules` 檔案在專案根目錄
2. 重新啟動 Cursor
3. 清除快取：設定 → 清除快取

### Q2: AI 回應太技術性，看不懂？
**解決方案：**
在 prompt 中明確要求：
```
請用產品經理能理解的語言說明，避免技術術語，多用比喻和實例。
```

### Q3: 分析結果不夠深入？
**解決方案：**
使用階段性分析：
```
第一階段：請先給我整體概覽
第二階段：深入分析 [具體功能]
第三階段：提供具體建議
```

### Q4: 生成的文檔品質不穩定？
**解決方案：**
1. 使用標準化的 prompt 模板
2. 明確指定輸出格式
3. 提供範例作為參考
