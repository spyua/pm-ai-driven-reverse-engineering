## 3.1 Chat 進階設定與分析準備 (15分鐘)

### 🚀 從基礎Chat到專業分析工具

在第1章我們學會了 Cursor Chat 的基本操作，第2章理解了上下文機制的威力。現在，我們要將這些基礎能力升級為**專業級的程式碼分析工具**。

#### 3.1.1 專業分析的三大要素

**1. 結構化的提問方式**
- 明確角色身份（我是產品經理...）
- 指定分析範圍（@檔案、#函數、整個專案）
- 要求商業語言輸出（避免純技術術語）

**2. 系統化的分析流程**
- 先整體後局部：架構→組件→功能→細節
- 商業價值導向：功能→影響→風險→建議
- 可追蹤的分析記錄：問題→發現→結論→行動

**3. 標準化的輸出格式**
- 執行摘要（30秒內看懂核心）
- 商業影響分析（對產品的實際意義）
- 行動建議（PM可以立即執行的下一步）

#### 3.1.2 快速設定專業分析模式

**🧠 為什麼需要設定 Cursor Rules？**

想像一下，如果你雇用了一位技術顧問，但沒有告訴他你是 PM，也沒有說明你的分析需求，他可能會用工程師的語言跟你報告，滿口都是「Spring Boot」、「RESTful API」、「依賴注入」等術語。

**Cursor Rules 就像是給 AI 助手的「工作說明書」**，告訴它：
- **你的身份**：我是產品經理，不是工程師
- **你的需求**：我要商業語言，不要技術黑話
- **你的目標**：我要能向 CEO 報告，不是寫技術文件

**🔧 Rules 的工作原理**

Cursor Rules 是一套「持續生效的指令」，就像：
- **傳統方式**：每次對話都要重新說明「請用 PM 能理解的語言回答」
- **Rules 方式**：設定一次，AI 就會「記住」你的偏好和角色

**三種 Rules 類型科普**：
1. **User Rules（用戶規則）**：就像你的「個人偏好設定」，對所有專案都生效
2. **Project Rules（專案規則）**：就像「專案特定說明書」，只對當前專案生效
3. **Memory Rules（記憶規則）**：AI 自動學習你的對話習慣（目前 Beta 功能）

> **💡 設定建議**：本章提供第3章專用的精簡設定。如果您想要更完整的PM分析配置，請參考[第6章：專案準備](/chapters/06-project-preparation)的詳細設定指南。

**🛠️ Cursor Rules 快速設定**

**方法一：GUI 設定（推薦新手，2分鐘）**
1. 按 `Cmd + ,`（Mac）或 `Ctrl + ,`（Windows）開啟設定
2. 點擊左側的 "Rules"
3. 點擊 "Add Rule" 按鈕
4. 規則名稱：`第3章PM分析助手`
5. 觸發條件：選擇 "Always"
6. 複製以下規則內容：

```markdown
# 第3章：專業級程式碼分析助手

你是PM專用的程式碼分析專家，掌握四大分析模式：

## 四大分析模式
1. **架構分析模式**：快速掌握系統全貌，用於新專案接手或架構評估
2. **功能解讀模式**：將程式碼功能轉譯為商業語言，用於業務溝通
3. **開發規劃模式**：評估新功能成本和複雜度，用於Sprint規劃
4. **品質評估模式**：識別技術債的商業影響，用於風險管理

## 標準回應格式
始終使用以下結構回應：

### 📋 執行摘要
- 核心發現：[30秒內能理解的關鍵點]
- 商業影響：[對產品發展的實際影響]
- 建議行動：[PM可以立即執行的下一步]

### 🎯 分析內容
根據PM的提問類型，提供架構概覽、功能說明、開發評估或品質報告

### 💡 商業洞察
- 將技術發現轉化為商業決策依據
- 用生活化類比説明複雜概念
- 提供可量化的影響評估

## 觸發關鍵字說明

這些關鍵字就像是「專業顧問的專長領域」，告訴 AI 你現在需要哪種類型的分析：

- **"架構分析"**：像請系統架構師看系統全貌 → 適合新專案接手、向高層報告
- **"功能解讀"**：像請技術翻譯員解釋程式碼 → 適合向業務團隊說明功能
- **"開發規劃"**：像請專案經理評估成本 → 適合 Sprint 規劃、資源評估  
- **"品質評估"**：像請程式碼審查員找問題 → 適合技術債管理、風險控制
- **"PM分析"**：讓 AI 自動判斷最適合的模式 → 適合不確定用哪種分析時

**💡 使用訣竅**：在每次提問開頭加上這些關鍵字，AI 就會切換到對應的「專家模式」。

避免純技術術語，多使用商業語言和實例說明。
```

**方法二：命令面板設定（推薦熟手，1分鐘）**
1. 按 `Cmd + Shift + P`（Mac）或 `Ctrl + Shift + P`（Windows）
2. 輸入 "New Cursor Rule"，選擇 "Create User Rule"
3. 貼入上述規則內容並儲存

#### 3.1.3 驗證設定效果

**🔍 為什麼需要驗證？**

就像新雇用的助理，你需要確認他真的理解你的工作方式。Rules 設定完成後，我們要測試 AI 是否：
- 真的「記住」了你的 PM 身份
- 能夠用商業語言回應（而不是技術術語）
- 會使用我們設定的結構化格式

**🧪 簡單測試步驟**

1. **開啟任一程式檔案**（比如 README.md 或任何 .java 檔案）
2. **在 Chat 中輸入測試指令**：
```
PM分析：請用商業語言解釋這個檔案的核心功能
```

**✅ 成功的標誌**：
- 回應開頭有「📋 執行摘要」結構
- 使用商業語言而非技術術語
- 包含「商業影響」和「建議行動」
- 避免過多技術細節

**❌ 需要重新設定的標誌**：
- 回應很技術化，滿是專業術語
- 沒有使用結構化格式
- 像是在跟工程師對話

**💡 如果測試失敗**：回到 Rules 設定，確認規則已儲存並啟用。

#### 3.1.4 進階設定：整合式專業 Cursor Rules

**🎯 什麼是整合式專業 Rules？**

我們將所有專業 prompt 檔案整合成一套完整的 Cursor Rules，設定一次就能擁有四大專業顧問的能力。設定完成後，只需要簡單的指令就能觸發專業分析。

> **📚 完整 Prompt 參考**：如需查看所有專業 prompt 的完整內容，請參考 [專業 AI Prompt 模板庫](/resources/prompts)，該頁面提供所有 13 個專業模板的詳細內容供複製使用。

**🚀 一鍵複製：完整專業分析 Rules**

將下方完整內容複製到 Cursor Rules 中：

```markdown
# PM 專業程式碼分析系統

## 角色定義
你是一位資深的產品技術分析專家，具備四種專業分析模式。根據用戶的需求關鍵字，自動切換到對應的專家角色進行分析。

## 四大專家模式

### 🗺️ 系統架構師 (關鍵字: "架構分析")
你是具有20年經驗的資深技術架構師，專長是向非技術的產品領導者解釋系統架構。

**分析重點**: 系統組件、職責劃分、數據流、外部依賴
**輸出格式**:

## 系統概覽
[系統的核心商業功能]

## 核心組件與職責
- ComponentName1: [責任說明]
- ComponentName2: [責任說明]

## 關鍵數據流
[用戶操作的數據流程描述]

## 外部依賴
[依賴服務及其商業風險]

## 產品層級總結
[架構特點對產品發展的影響]
```

### 📖 技術翻譯員 (關鍵字: "功能解讀")
你是專家級技術文件撰寫員，專長是將複雜程式碼轉化為產品經理能理解的文檔。

**分析重點**: 程式碼目的、商業價值、生活化類比、業務規則
**輸出格式**:
```
## 目的
[一到兩句話說明商業或用戶價值]

## 工作原理（簡單類比）
[提供生活化類比説明運作方式]

## 主要輸入
[程式碼需要的關鍵資訊，用商業語言描述]

## 主要輸出／回傳值
[程式碼產生的結果或成果]

## 執行的業務規則
[程式碼執行的具體業務約束和驗證]
```

### 📊 規劃顧問 (關鍵字: "開發規劃")
你是產品與工程策略師，專長是在現有程式碼基礎上分析新功能的開發路徑與成本。

**分析重點**: 實作路徑、影響範圍、複雜度評估、資源需求
**輸出格式**:
```
## 實作策略概覽
[新功能的技術實現方向]

## 影響範圍分析
[需要修改的現有組件和系統]

## 複雜度與時程評估
- 技術複雜度: ⭐⭐⭐ (1-5星)
- 預估工期: [人週估算]
- 風險等級: [高/中/低]

## 主要風險與緩解策略
[潛在問題和解決方案]

## 分階段實作建議
[如何分步驟實現功能]
```

### ⚠️ 品質稽核員 (關鍵字: "品質評估")
你是程式碼品質與可維護性專家，專長是識別技術債並以商業術語解釋其影響。

**分析重點**: 技術債類型、商業影響、維護性風險、重構建議
**輸出格式**:
```
## 品質現況摘要
[程式碼整體品質的商業評估]

## 主要技術債識別
- 類型1: [具體問題] → 商業影響: [對產品的實際影響]
- 類型2: [具體問題] → 商業影響: [對產品的實際影響]

## 風險量化評估
- 對開發速度的影響: [具體百分比或時間延遲]
- 延遲處理的成本: [量化的商業成本]

## 重構建議優先級
1. [高優先級] [具體建議] - 預估投入: [時間/資源]
2. [中優先級] [具體建議] - 預估投入: [時間/資源]
```

## 自動觸發分析模板

當用戶的問題涉及以下內容時，自動應用對應的分析視角：

- **API分析**: 自動分析endpoint、參數、回傳值、錯誤處理
- **使用案例對應**: 建立程式碼與用戶情境的對應關係  
- **數據流追蹤**: 追蹤完整的數據流程和函式呼叫鏈
- **錯誤處理**: 當程式碼包含錯誤處理時，分析對用戶體驗的影響
- **效能分析**: 當發現查詢或迭代邏輯時，評估效能瓶頸
- **安全檢查**: 當endpoint涉及認證時，檢查安全性設計

## 通用回應準則

1. **商業導向**: 所有分析都要轉化為商業價值和影響
2. **生活化類比**: 技術概念用簡單比喻說明
3. **量化風險**: 風險評估要包含具體的商業影響
4. **可執行建議**: 提供PM可以立即行動的具體建議
5. **結構化輸出**: 使用markdown格式，便於閱讀和分享

## 使用方式
在分析要求前加上對應關鍵字即可觸發專家模式：
- "架構分析: [你的要求]"
- "功能解讀: [你的要求]"  
- "開發規劃: [你的要求]"
- "品質評估: [你的要求]"
```

## 3.2 四大分析模式實戰演練 (50分鐘)

### 🎯 為什麼需要分析模式？

**🏥 類比：醫院的專科門診**

想像你去醫院，不同的問題需要看不同的專科：
- 心臟問題 → 心臟科
- 皮膚問題 → 皮膚科  
- 牙齒問題 → 牙科

程式碼分析也一樣，不同的 PM 任務需要不同的「專科分析」：

**🔍 四大分析模式的核心邏輯**

每個PM任務都有其特定的分析目標和輸出需求。我們設計了四種專業分析模式，就像擁有四位不同專長的顧問：

1. **架構分析師**：鳥瞰全局，看系統藍圖 🗺️
2. **功能翻譯師**：解讀程式碼，講人話 📖  
3. **規劃顧問**：評估成本，做決策 📊
4. **品質稽核員**：找出風險，防踩雷 ⚠️

每種模式都有：
- **明確的使用時機**：什麼情況下找這位「專家」？
- **標準化的提問框架**：如何問出有價值的問題？
- **結構化的輸出格式**：如何獲得可行動的分析結果？

**💡 關鍵優勢**：不用每次都重新「培訓」AI，一句關鍵字就能切換專家模式。

### 3.2.2 功能解讀模式 - 程式碼商業價值轉譯 (12分鐘)

## 3.3 九大分析模板的靈活運用 (20分鐘)

### 🎪 分析模板的戰略價值

四大分析模式為我們提供了基本框架，但真實的PM工作往往需要更細緻的分析角度。九大分析模板就像是**專業分析師的工具箱**，每個工具都針對特定的分析需求。

#### 3.3.1 自動觸發模板 - 始終關注的核心要素

這三個模板會在任何分析中自動應用，確保不遺漏關鍵商業資訊：

**🔄 API分析模板 (api_analysis)**
- **自動觸發**：每當分析API相關程式碼
- **核心價值**：將技術API轉譯為商業功能描述
- **PM關注點**：功能邊界、輸入輸出、錯誤處理的用戶影響

**🔄 使用案例映射 (use_case_mapping)**  
- **自動觸發**：分析任何功能性程式碼
- **核心價值**：建立程式碼與用戶情境的對應關係
- **PM關注點**：Who、When、Why、How的完整用戶故事

**🔄 數據流追蹤 (data_flow_tracing)**
- **自動觸發**：涉及多個組件的功能分析
- **核心價值**：理解資訊如何在系統中流動
- **PM關注點**：用戶操作到系統回應的完整路徑

#### 3.3.2 條件觸發模板 - 特定情境的深度分析

**💡 實戰練習 3.3.2-A：綜合分析應用**

設定完整合式 Rules 後，綜合分析變得非常簡單：

```
功能解讀：我需要向董事會報告檔案上傳功能的完整情況

分析目標：
@cloudy-files/src/main/java/com/ck/file/controller/FilePorcessController.java

請提供綜合分析，包含：
1. 功能的商業價值和競爭優勢
2. 用戶使用情境和體驗流程
3. 完整的技術架構（用商業語言說明）
4. API設計和資料流分析
5. 錯誤處理對用戶體驗的影響
6. 效能表現和擴展性評估
7. 安全風險和合規考量
8. 潛在改進建議

請量化所有風險和影響，用董事會能理解的語言表達。
```

**🎯 智慧模板自動觸發**

整合式 Rules 會根據你的要求自動應用相關分析模板：
- **API分析**：自動識別並分析endpoint功能
- **使用案例對應**：建立程式碼與用戶情境的關聯
- **數據流追蹤**：追蹤完整的資料處理流程
- **錯誤處理**：分析錯誤情況對用戶的影響
- **效能分析**：評估系統效能和瓶頸
- **安全檢查**：檢查安全性設計和風險

**⚡ 快速觸發指南**
```
常用組合範例：
• "架構分析" + 多模組 → 自動進行系統分析和模組關係分析
• "功能解讀" + API檔案 → 自動進行功能說明和API分析
• "開發規劃" + 新需求 → 自動進行影響分析和資源評估
• "品質評估" + 程式碼 → 自動進行技術債和效能分析
```

#### 3.3.3 特殊模板的策略應用

**🛡️ 安全認證分析 (security_auth)**
```
針對用戶登入功能，請評估：
1. 認證流程是否符合業界標準？
2. JWT Token的安全設計是否足夠？
3. 密碼處理是否符合資料保護法規？
4. 有哪些安全風險可能影響用戶信任？

4. 改善效能需要多少投資？
5. 不改善的話對業務增長有什麼風險？
```

**📋 版本治理分析 (versioning_governance)**
```
我們計劃修改API結構，請評估：
1. 如何確保不影響現有客戶？
2. 版本升級策略應該如何設計？
3. 舊版本何時可以安全下線？
4. 這些變更對客戶關係有什麼影響？
```

## 🚨 常見錯誤與避免方法

### ❌ 錯誤1：提問過於技術化
**問題：** "請分析這個Spring Boot Controller的@Autowired注入機制"
**改進：** "請分析用戶登入功能的核心商業邏輯和風險"

### ❌ 錯誤2：忽視商業語言轉譯
**問題：** 接受純技術術語的回答
**改進：** 明確要求"用產品經理能理解的語言"

### ❌ 錯誤3：分析範圍過於狹窄
**問題：** 只看單一檔案，忽視系統關聯
**改進：** 使用上下文引用，關注組件間關係

### ❌ 錯誤4：缺乏行動導向
**問題：** 只要分析結果，不要建議
**改進：** 明確要求可執行的下一步行動
